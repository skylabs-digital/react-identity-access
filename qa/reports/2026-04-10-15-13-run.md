## Critical & High findings

### 🔴 CRITICAL-1 · `javascript:` accepted as `baseUrl`

**Flow**: [auth-config-fuzz](.tmp/auth-config-fuzz.json) · vector `baseUrl="javascript:alert(1)"`

The library constructs a `SessionManager` with `baseUrl: "javascript:alert(1)"` without validation. Any downstream code that interpolates `baseUrl` into a navigation sink (router, `window.location.assign`, `<a href>`) would execute attacker-controlled JavaScript. Same class of issue as the `_auth` URL transfer we removed in v2.31.

**Fix**: validate `baseUrl` scheme at construction time — allow only `http:` / `https:`. Reject `javascript:`, `data:`, `vbscript:`, `file:` with a typed error. Apply the same rule in `AppProvider`, `AuthProvider` standalone, and any other public entry-point.

**Crystallize**: [`qa/flows/edge/baseurl-scheme-validation.md`](../flows/edge/baseurl-scheme-validation.md) — asserts the constructor throws on dangerous schemes.

### 🟠 HIGH-2 · Negative `refreshQueueTimeout` silently accepted

**Flow**: [auth-config-fuzz](.tmp/auth-config-fuzz.json) · vector `refreshQueueTimeout=-1`

Construction succeeds with a negative timeout. Downstream this either causes `setTimeout` to clamp to `0ms` (instant timeout, breaks the queue) or throws deep in the refresh loop without a typed error.

**Fix**: validate numeric config values (`refreshQueueTimeout`, `proactiveRefreshMargin`) are finite non-negative numbers. Reject `-1`, `NaN`, `Infinity`, strings. Throw `ConfigurationError` at construction.

---

### 🟠 HIGH-3 to HIGH-7 · `setTokens()` accepts structurally-broken JWTs

**Flow**: [malformed-token-fuzz](.tmp/malformed-token-fuzz.json) · vectors `malformed-base64-body`, `two-segments`, `four-segments`, `truncated`, `non-json-header`

`SessionManager.setTokens()` trusts the caller completely. The library never parses the JWT; it uses the caller-supplied `expiresIn` to compute a local expiry and persists the token bytes unchanged. A caller passing an empty string, a 2-segment JWT, or a truncated blob produces `hasValidSession() === true`.

**Impact**: not directly exploitable (the caller is trusted in this library's threat model), but it means:
- `AuthApiService.login()` bugs that build malformed tokens are invisible until the first API call fails.
- Tooling that restores a session from storage after tampering (see V2 in storage-quota-fuzz) doesn't catch rot.
- The error surface is `"Refresh token is invalid"` on the first refresh, which is confusing because the access token was malformed, not the refresh token.

**Fix** (two options):
1. **Strict** — parse the JWT on `setTokens`, reject if segments ≠ 3 or header is not parseable JSON.
2. **Lenient** — keep trust-the-caller semantics, but document it loudly in `docs/api-reference.md` and surface a more precise error on refresh failure.

**Crystallize**: [`qa/flows/edge/settokens-structural-validation.md`](../flows/edge/settokens-structural-validation.md).

---

### 🟠 HIGH-8 / HIGH-9 · `expiresIn=0` and `expiresIn=NaN` accepted

**Flow**: [malformed-token-fuzz](.tmp/malformed-token-fuzz.json) · vectors `expiresIn-zero`, `expiresIn-nan`

`setTokens({ expiresIn: 0 })` → `hasValidSession() === true` with an expiry of `Date.now()`. `expiresIn: NaN` → expiry is `Date.now() + NaN * 1000` → the computed timestamp is `NaN`, and `hasValidSession()` still returns `true` (presumably the expiry comparison short-circuits). Behavior is non-deterministic and leaks into the proactive refresh scheduler.

**Fix**: clamp `expiresIn` in `setTokens`: require `Number.isFinite(expiresIn) && expiresIn > 0`, otherwise reject.

---

### 🟠 HIGH-10 · `QuotaExceededError` on `setTokens` propagates raw with no fallback

**Flow**: [storage-quota-fuzz](.tmp/storage-quota-fuzz.json) · vector `V1-raw-quota-propagation`

`setTokens()` catches nothing when `localStorage.setItem` throws. The raw DOM `QuotaExceededError` bubbles out of the library. There is no in-memory fallback: `hasValidSession()` returns `false` and the session is effectively lost.

**Fix**: wrap storage writes in `try { } catch (QuotaExceededError) { fallBackToMemory() }`. Emit a typed `StorageUnavailableError` through `onSessionExpired` so callers can respond. Document the in-memory fallback semantics.

---

### 🟠 HIGH-11 · Hostile storage (always throws) surfaces raw error

**Flow**: [storage-quota-fuzz](.tmp/storage-quota-fuzz.json) · vector `V6-hostile-storage-propagation`

When `localStorage` throws on every operation (e.g., Safari private mode, storage disabled by policy), both `setItem` and `getItem` errors bubble raw to the caller (`"storage is disabled"`). The library should fall back to in-memory storage and log a warning, not crash.

**Fix**: in the storage adapter, probe `localStorage` at construction (`setItem('__probe__', '1'); removeItem('__probe__')`) and fall back to an in-memory Map if it throws. Same `StorageUnavailableError` callback as HIGH-10.

## Recurrent patterns

1. **No input validation layer** (drives CRITICAL-1, HIGH-1, HIGH-2, HIGH-3..9, 7 medium)
   Config, tokens, and storage records are trusted. Single fix would clear ~60% of findings.

2. **Raw DOM errors bubble from the storage layer** (HIGH-10, HIGH-11)
   Library does not abstract storage failures into typed errors. Callers get `QuotaExceededError` / `"storage is disabled"` with no recovery path.

3. **Proactive scheduler is naive about `Date.now()` jumps** (1 medium)
   Scheduler uses a single `setTimeout`; backward clock jumps cause it to miss its window. The on-demand path catches it on next API call, so users rarely notice — but for idle sessions it means silent token-expiry risk.

4. **Doc-reality divergence in `security.md`** (1 medium + 2 low, magic-link-dedup)
   `docs/security.md` claims `AuthProvider.sendMagicLink` dedups identical calls; black-box probe at the `AuthApiService` layer saw no dedup. Either fix the docs or the code.
